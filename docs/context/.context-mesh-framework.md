# Context Mesh Framework

## Overview
Context Mesh organizes durable project knowledge in three steps:

1. **Intent**
   Record what the project does and why it matters.
2. **Build**
   Implement changes while respecting documented decisions and patterns.
3. **Learn**
   Preserve outcomes, tradeoffs, and reusable knowledge after meaningful changes.

This repository documents an existing Kotlin/JVM text RPG. The codebase remains the primary source of truth; Context Mesh exists to preserve only the information that is worth reloading later.

## File Type Separation Rules

### Feature Files
Location: `context/intent/feature-*.md`

Feature files contain only:
- What the feature does
- Why it exists
- Acceptance criteria

Feature files must not contain:
- Technical implementation details
- Library/framework choices
- Architecture explanations
- Code examples

### Decision Files
Location: `context/decisions/*.md`

Decision files contain:
- The technical choice that was made
- Context and constraints
- Rationale
- Alternatives considered
- Outcomes over time

Decision files explain **HOW the project chose to solve something**.

### Pattern Files
Location: `context/knowledge/patterns/*.md`

Pattern files contain:
- Reusable implementation guidance
- Structure/convention explanations
- Real examples from the repository
- When to use the pattern

Pattern files explain **HOW TO IMPLEMENT recurring solutions**.

### Anti-Pattern Files
Location: `context/knowledge/anti-patterns/*.md`

Anti-pattern files are optional and should exist only for repeated, risky, or costly mistakes worth avoiding.

### Agent Files
Location: `context/agents/`

Agent files are optional and should be created only when the repository needs specialized agent roles or workflows.

## Selective Loading Rule
Context Mesh must be used as selective memory.

Required behavior:
- Do not bulk-load the entire `context/` tree.
- Load only the smallest set of files relevant to the task.
- Prefer code and data files first for local debugging, balance, and narrow refactors.
- Use Context Mesh mainly for cross-session continuity, architecture, durable patterns, and partial history.

Good uses:
- Broad refactors
- Architecture or data model changes
- Resuming work after a gap
- Handing work off
- Checking whether a previous technical decision exists

Bad uses:
- Loading every feature/decision/pattern file before simple edits
- Using `context/` instead of reading the actual source files
- Writing long duplicate summaries of code that can be rediscovered cheaply

## Context Compaction And Checkpoint Rule
Before compacting the IDE/context window or handing work off:
- Save useful, durable information into `/context`
- Prefer the most specific existing file first
- Keep entries concise and high-signal

Checkpoint priority:
1. Relevant feature file if user-visible behavior changed
2. Relevant decision file if a technical choice changed
3. Relevant pattern file if a reusable pattern emerged
4. `context/evolution/changelog.md` for important milestone/state changes
5. `context/evolution/working-memory.md` only for concise partial history that matters later and does not fit better elsewhere

Do not checkpoint:
- Temporary speculation
- Command spam
- Long code summaries
- Details that are obvious from the modified files

## Plan, Approve, Execute

### 1. Plan
- Decide whether `context/` is needed for the task
- Load only relevant context files
- Identify whether the work changes Intent, Build, or Learn artifacts

### 2. Approve
- Check whether an existing decision or pattern already covers the work
- If a new durable technical choice is introduced, create or update a decision file
- If user-facing behavior changes materially, update the relevant feature file

### 3. Execute
- Implement the change
- Keep features, decisions, and patterns separate
- Update changelog and any necessary checkpoint notes

## When To Create A File
Create a new file only when the knowledge is durable and likely to be useful later:
- New user-facing capability or module: feature file
- New technical or architectural choice: decision file
- New reusable implementation convention: pattern file
- Repeated harmful shortcut: anti-pattern file

## When Not To Create A File
Do not create a new file for:
- Tiny local edits
- One-off debugging notes
- Duplicate summaries
- Transient implementation details

Prefer updating an existing file when the knowledge fits there.

## Context Structure
```text
context/
|-- .context-mesh-framework.md
|-- intent/
|   |-- project-intent.md
|   `-- feature-*.md
|-- decisions/
|   `-- 00x-*.md
|-- knowledge/
|   |-- patterns/
|   `-- anti-patterns/
|-- agents/
`-- evolution/
    |-- changelog.md
    `-- working-memory.md
```

## Bidirectional Links Rules
- Feature files must link to relevant decisions
- Feature files may link to relevant patterns
- Decision files should link back to supported features or related patterns
- Pattern files should link to related decisions and features when applicable
- Prefer relative markdown links within `context/`

## Status Section Requirements
Feature, decision, and pattern files should end with a status section containing at least:
- **Created**
- **Status**

Optional but encouraged:
- **Phase**
- **Note**

## Definition Of Done
Before considering work complete:
- Only relevant context files were loaded
- Durable new knowledge was written back into `context/` if needed
- Features, decisions, and patterns remain separated
- Links between related files are valid where applicable
- `context/evolution/changelog.md` reflects important project-state changes
